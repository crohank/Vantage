# Financial Query Annotation Guidelines

## Purpose

This document defines the annotation schema for classifying user queries to the Financial Research Analyst system. The annotations are used to evaluate query resolution accuracy and agent routing.

## Intent Categories

### 1. `stock_analysis`
**Definition:** The user wants a comprehensive analysis of a specific stock, including price data, risk assessment, scenario modeling, and an investment memo.

**Indicators:**
- Mentions a specific company or ticker
- Uses words like "analyze", "report", "outlook", "how is X doing"
- Asks "should I buy/sell/hold" for a specific stock
- Requests general information about a specific company's stock

**Examples:**
- "Analyze Tesla stock" → stock_analysis
- "How is Apple doing?" → stock_analysis
- "Should I buy MSFT?" → stock_analysis
- "What's the outlook for Nvidia?" → stock_analysis

### 2. `risk_assessment`
**Definition:** The user specifically asks about risk factors, downside potential, volatility, or safety of a specific stock.

**Indicators:**
- Uses words like "risk", "risky", "volatile", "downside", "bear case", "safety", "what could go wrong"
- Focuses on negative scenarios or protective analysis

**Examples:**
- "What are the risks of investing in NVDA?" → risk_assessment
- "How volatile is COIN?" → risk_assessment
- "Give me a bear case for Tesla" → risk_assessment
- "How safe is KO as a dividend stock?" → risk_assessment

**Edge case:** "Should I buy AAPL?" is `stock_analysis`, not `risk_assessment`, because the user wants a full recommendation, not just risk info.

### 3. `comparison`
**Definition:** The user wants to compare two or more stocks, usually to decide between them.

**Indicators:**
- Mentions two or more companies/tickers
- Uses words like "compare", "vs", "versus", "which is better", "or"

**Examples:**
- "Compare AAPL and MSFT" → comparison
- "NVDA vs AMD" → comparison
- "Which is better, Google or Meta?" → comparison

**Note:** For `expected_ticker`, use the first mentioned stock.

### 4. `market_outlook`
**Definition:** The user asks about general market conditions, sectors, or macroeconomic trends without referencing a specific stock.

**Indicators:**
- No specific company/ticker mentioned
- Asks about "the market", sectors, interest rates, inflation
- Uses words like "outlook", "forecast", "performing"

**Examples:**
- "How is the tech sector doing?" → market_outlook
- "What's the market outlook?" → market_outlook
- "Is the market overvalued?" → market_outlook
- "What should I buy right now?" → market_outlook (no specific stock)

**Edge case:** "How does Fed rate policy affect Apple stock?" → `stock_analysis` because a specific ticker is mentioned. The macro context is secondary.

### 5. `document_lookup`
**Definition:** The user asks about specific information from SEC filings, annual reports, or other financial documents.

**Indicators:**
- References "10-K", "10-Q", "annual report", "filing", "SEC"
- Asks what management or the company "said" about something
- Requests specific sections like "risk factors", "MD&A"

**Examples:**
- "What did Apple's 10-K say about litigation?" → document_lookup
- "Show me Tesla's risk factors from their annual report" → document_lookup

## Annotation Fields

| Field | Type | Description |
|-------|------|-------------|
| `id` | int | Unique identifier |
| `query` | string | The user's input text |
| `intent` | string | One of the 5 categories above |
| `expected_ticker` | string/null | Expected resolved ticker, or null if no specific stock |
| `complexity` | string | "simple" (single entity, clear intent) or "multi-hop" (multiple entities, implied parameters, or ambiguous) |
| `source` | string | "manual" (human-written) or "llm_generated" |
| `human_verified` | boolean | Whether a human has verified this annotation |
| `notes` | string | Optional notes about edge cases or reasoning |

## Decision Tree for Ambiguous Cases

```
Does the query mention a specific stock/company?
├── NO → market_outlook
└── YES
    ├── Does it ask about SEC filings/documents?
    │   ├── YES → document_lookup
    │   └── NO → continue
    ├── Does it compare 2+ stocks?
    │   ├── YES → comparison
    │   └── NO → continue
    ├── Does it focus specifically on risk/downside/volatility?
    │   ├── YES → risk_assessment
    │   └── NO → stock_analysis
```

## Quality Review Process

1. **Generation:** Queries can be written manually or generated by LLM
2. **Labeling:** Apply intent category using the decision tree above
3. **Verification:** All LLM-generated annotations must be human-verified
4. **Edge cases:** Document reasoning in the `notes` field
5. **Consistency:** Re-check annotations when taxonomy is updated
